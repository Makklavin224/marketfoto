Create worker/tasks.py with the remove_background_job function. This is the RQ job that runs in a forked subprocess.

CRITICAL per D-03, D-07, UPLD-12, and PITFALLS.md Pitfall 1:
- Create rembg session ONCE at module level: `from rembg import new_session; rembg_session = new_session("birefnet-general")` -- this is loaded when the worker process forks, reused across all jobs in that fork. The module-level session is the documented pattern for rembg memory management.
- The worker Dockerfile already pre-downloads the model. The Docker --max-jobs 100 (already in docker-compose.yml) handles recycling.

Function signature: `def remove_background_job(image_id: str, user_id: str) -> None`

Implementation steps:
1. Create a SYNCHRONOUS SQLAlchemy engine (not async -- RQ workers run sync code). Use `create_engine()` from sqlalchemy, NOT the async engine from database.py. Connect to same database_url from environment (os.environ.get or import settings).
2. Create a synchronous MinIO client using `minio.Minio(endpoint, access_key, secret_key, secure=False)` with settings from environment.
3. Record start_time = time.time().
4. Update image status to 'processing' in DB (UPDATE images SET status='processing' WHERE id=image_id).
5. Download original image from MinIO: `minio_client.get_object("originals", f"{user_id}/{image_id}.{ext}")`. The original_url field contains the MinIO path -- parse it to extract the object key, or construct from user_id/image_id pattern. Read the image bytes.
6. Open with Pillow, call `rembg.remove(input_image, session=rembg_session)` passing the pre-initialized session. The result is a PIL Image with alpha channel (RGBA).
7. Save result as PNG to a BytesIO buffer.
8. Upload to MinIO processed bucket: `minio_client.put_object("processed", f"{user_id}/{image_id}.png", data, length, content_type="image/png")`.
9. Calculate processing_time_ms = int((time.time() - start_time) * 1000).
10. Update image record: status='processed', processed_url=f"processed/{user_id}/{image_id}.png", processing_time_ms=processing_time_ms.

Timeout handling per D-04, UPLD-09:
- RQ supports job timeout. The endpoint (Task 2) will enqueue with `job_timeout=30`. If the job exceeds 30s, RQ kills the forked process.
- Add an `on_failure` callback or wrap the job in try/except to catch any exception and update status='failed', error_message to the DB. Use a separate try/except around the entire function body. In the except block: update image status='failed', error_message=str(exception)[:500]. For timeout specifically, RQ raises `rq.timeouts.JobTimeoutException`.
- Also add a finally-like pattern: if exception occurs, ensure status is set to 'failed'.

Error handling:
- If MinIO download fails: status='failed', error_message='Failed to download original image'
- If rembg fails: status='failed', error_message='Background removal failed: {str(e)}'
- If MinIO upload fails: status='failed', error_message='Failed to save processed image'

Do NOT use async/await anywhere in this file. The worker runs in synchronous forked subprocesses.

Create backend/app/api/images.py with an APIRouter(prefix="/api/images", tags=["images"]):

**Endpoint 1: POST /api/images/{image_id}/remove-background** (per D-01, UPLD-05)
- Requires auth (Depends on get_current_user -- this will exist from Phase 2. For now, create a placeholder dependency that extracts user_id. Add a comment: "# TODO: Replace with real auth dependency from Phase 2").
- Accept image_id as UUID path parameter.
- Query DB for image record: SELECT from images WHERE id=image_id.
- If not found: return 404 {"error": "Image not found"}.
- If image.user_id != current_user.id: return 404 (don't leak existence).
- If image.status == 'processing': return 409 {"error": "Already processing"} (per D-06, UPLD-11 from Phase 3).
- If image.status == 'processed': return 409 {"error": "Already processed"}.
- Connect to Redis, get RQ Queue: `from redis import Redis; from rq import Queue; q = Queue(connection=Redis.from_url(settings.redis_url))`.
- Enqueue: `q.enqueue("worker.tasks.remove_background_job", str(image_id), str(image.user_id), job_timeout=30)` -- the 30-second timeout per D-04/UPLD-09. Use string function path since worker is a separate process.
- Return 202 with RemoveBackgroundResponse(id=str(image.id), status="processing").

IMPORTANT: The endpoint does NOT update status to 'processing' itself. The worker job does that at the start of execution. This avoids a race where the status is set to 'processing' but the job never starts (e.g., Redis queue full). However, for immediate UX feedback, set status='processing' in the endpoint before enqueue via DB update. This is acceptable because if the job fails to enqueue, the user will see 'processing' until timeout.

**Endpoint 2: GET /api/images/{image_id}/status** (per D-05, UPLD-08)
- Requires auth (same placeholder dependency).
- Accept image_id as UUID path parameter.
- Query DB for image record.
- If not found or user_id mismatch: return 404.
- Return 200 with ImageStatusResponse containing status, processed_url (generate a presigned GET URL from MinIO if processed_url exists, valid for 1 hour), error_message, processing_time_ms.
- For the presigned URL: use `minio_client.presigned_get_object("processed", image.processed_url, expires=timedelta(hours=1))` if status is 'processed'. Otherwise processed_url is None.

Create a helper function `get_minio_client()` that returns a Minio client instance, reusable across endpoints.
