<objective>
Build the YooKassa payment creation and webhook processing backend for MarketFoto.

Purpose: Users need to pay for subscriptions (starter/business monthly/annual) and one-time card purchases. YooKassa is the payment processor. This plan creates the payment initiation flow (user clicks pay -> redirect to YooKassa) and the webhook receiver (YooKassa confirms payment -> update user plan/credits). Webhook idempotency and IP whitelist are critical for payment security.

Output: Payment schemas, service layer with YooKassa SDK, and API router with POST /create and POST /webhook endpoints.
</objective>

3. **`check_ip_whitelist(ip: str) -> bool`** function using `ipaddress` module (per D-05):
   Check if request IP is in YOOKASSA_IP_RANGES. Return True if allowed, False otherwise.

4. **`create_yookassa_payment(user_id: UUID, variant: str, return_url: str) -> tuple[Payment, str]`** (per D-01, D-10):
   - Configure YooKassa SDK: `Configuration.account_id = settings.yookassa_shop_id`, `Configuration.secret_key = settings.yookassa_secret_key`
   - Look up variant in PLAN_CONFIG
   - Call `yookassa.Payment.create({...})` with amount in RUB (divide kopecks by 100 for value string, e.g., "499.00"), currency "RUB", confirmation type "redirect" with return_url, description from config, metadata with user_id and variant
   - Create Payment record in DB with status="pending", yookassa_payment_id from response, yookassa_confirmation_url from response.confirmation.confirmation_url
   - Return (payment_record, confirmation_url)

5. **`process_webhook(payload: dict) -> bool`** (per D-03, D-04, D-05, D-11):
   - Extract event type and payment object from payload
   - Only process "payment.succeeded" events; ignore others, return True
   - Extract yookassa_payment_id from payload["object"]["id"]
   - **Idempotency check**: SELECT FROM payments WHERE yookassa_payment_id = ? AND status = 'succeeded'. If found, return True (already processed).
   - Update Payment record: status = "succeeded"
   - Extract variant from payment metadata
   - Look up PLAN_CONFIG[variant]
   - If type == "subscription" (per D-04):
     - `UPDATE users SET plan = ?, credits_remaining = ?, subscription_expires_at = now() + interval '? days', subscription_yookassa_id = ? WHERE id = ?`
     - Map plan names: "starter_annual" -> "starter", "business_annual" -> "business" (the user.plan column only allows 'free'/'starter'/'business')
   - If type == "one_time" (per D-05):
     - Atomic credit addition: `UPDATE users SET credits_remaining = credits_remaining + 1 WHERE id = ?`
     - Do NOT change plan
   - Return True on success

IMPORTANT: For the atomic credit deduction pattern referenced in D-11, that will be used by the render endpoint (Phase 7). In this phase, we implement the credit ADDITION (from payments) and plan updates. The atomic deduction SQL pattern is: `UPDATE users SET credits_remaining = credits_remaining - 1 WHERE id = :uid AND credits_remaining > 0 RETURNING credits_remaining`. Document this in a comment in the service file for Phase 7's reference.

Use `from yookassa import Configuration, Payment as YooKassaPayment` to avoid name collision with the SQLAlchemy Payment model.

Create `backend/app/api/payments.py` with router prefix="/api/payments", tags=["payments"]:

1. **POST /create** (per D-01, D-03):
   - Depends on `get_current_user`
   - Accepts `CreatePaymentRequest` body
   - Calls `create_yookassa_payment(user.id, request.variant, f"{settings.app_url}/payment/success")`
   - Returns `CreatePaymentResponse(payment_id=payment.id, confirmation_url=confirmation_url)`
   - On YooKassa SDK error, return 502 with `{"error": "Payment service unavailable"}`

2. **POST /webhook** (per D-03, D-05):
   - NO authentication (YooKassa sends webhooks without JWT)
   - Get client IP from `request.client.host` (also check X-Forwarded-For header if behind nginx: `request.headers.get("x-forwarded-for", "").split(",")[0].strip() or request.client.host`)
   - Call `check_ip_whitelist(client_ip)`. If False, return 403 `{"error": "Forbidden"}`
   - Accept raw JSON body (`request: Request` from FastAPI, parse with `await request.json()`)
   - Call `process_webhook(payload)`. Return 200 `{"status": "ok"}` immediately (per Pitfall 9: respond 200 fast, process synchronously for MVP simplicity since webhook processing is just DB writes, not heavy compute)
   - On any exception during processing, log the error but still return 200 to prevent YooKassa retries for transient errors. Log the full payload for manual reconciliation.
