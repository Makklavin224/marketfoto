Add to `backend/app/api/payments.py` two new endpoints:

1. **GET /history** (per D-06, PAY-08):
   - Depends on `get_current_user`
   - Accepts optional query params: `limit: int = 20`, `offset: int = 0`
   - Query: `SELECT * FROM payments WHERE user_id = ? ORDER BY created_at DESC LIMIT ? OFFSET ?`
   - Also query total count: `SELECT count(*) FROM payments WHERE user_id = ?`
   - Return `PaymentHistoryResponse(payments=[PaymentResponse.model_validate(p) for p in results], total=count)`

2. **POST /cancel-subscription** (per D-07, PAY-09):
   - Depends on `get_current_user`
   - Check user.plan != 'free', otherwise return 400 `{"error": "No active subscription"}`
   - Check user.subscription_expires_at is not None, otherwise return 400
   - Clear `user.subscription_yookassa_id = None` (cancels auto-renewal intent)
   - Do NOT change user.plan or credits_remaining -- plan stays active until expires_at
   - Return `CancelSubscriptionResponse(message="Subscription cancelled. Your plan is active until {expires_at}", plan_active_until=user.subscription_expires_at)`
   - Log cancellation for audit

Note: In a full production system, we would also call YooKassa API to cancel the recurring payment. For MVP, clearing subscription_yookassa_id is sufficient since the cron job (Task 2) handles expiration, and we haven't implemented autopay yet (requires YooKassa manager approval per PITFALLS.md Pitfall 9). Add a TODO comment referencing this.

Create `backend/app/services/subscription_cron.py` as a standalone runnable script (per D-08, PAY-10):

The script should:
1. Be executable via `python -m app.services.subscription_cron` from the backend directory
2. Use asyncio to run the async DB operations
3. Implement `async def expire_subscriptions()`:
   - Query: `SELECT id, email, plan, subscription_expires_at FROM users WHERE subscription_expires_at < now() AND plan != 'free'`
   - For each expired user:
     - `UPDATE users SET plan = 'free', credits_remaining = 3, subscription_expires_at = NULL, subscription_yookassa_id = NULL WHERE id = ?`
     - Log: `f"Expired subscription for user {user.email}: {user.plan} -> free"`
   - Return count of expired users
4. Include `if __name__ == "__main__":` block that runs `expire_subscriptions()` via `asyncio.run()`
5. Print summary: `f"Expired {count} subscription(s)"`

Use synchronous SQLAlchemy engine for the cron script (simpler for standalone execution):
- Create a separate sync engine: `create_engine(settings.database_url)` (not async)
- Use `Session(engine)` for sync queries
- This avoids asyncio complexity in a simple cron script

Add a comment at the top of the file explaining how to set up the cron job:

```python
# Run daily via cron:

#
# Or via Docker:
# docker compose exec backend python -m app.services.subscription_cron
```

Claude's discretion on cron implementation: Using a standalone script invoked by system cron (or docker compose exec) is simplest for MVP. APScheduler or RQ Scheduler add dependencies and complexity. System cron is battle-tested and appropriate for a daily job.
